# 🏗️ 인테리어 자재 단가 리서치 스킬 (사무실 전용)

> **목적:** 매주 반복하는 "자재 단가 웹 리서치 → md 정리 → 노션 공유"를 한 번 박제해 두고, "리서치 돌려줘" 한마디 또는 예약 실행으로 매번 같은 절차·같은 포맷으로 재현한다.

핵심 파이프라인:
`카테고리·키워드 확인 → 브라우저 MCP로 출처 3곳+ 탐색 → 단가 비교 md 정리(전주 대비 포함) → 노션 업로드 → interior-cost 카탈로그 반영 → 로컬 저장`

## 입력 (인자)

- **인자 없음** — 기본 카테고리 세트로 진행하되, 대화 중이면 먼저 카테고리·키워드를 **사용자에게 묻는다**.
- **카테고리/키워드 지정** — 예: `리서치 돌려줘 바닥재 강마루`, `자재 단가 욕실 타일·수전`
- **`cron` / 무인 실행** — 사용자에게 묻지 않고, **리서치 큐의 다음 배치(기본 25건)** 를 조사한다(아래 "리서치 큐 배치" 참조). 큐를 조회할 수 없을 때만 "기본 카테고리 세트"로 폴백.

### 리서치 큐 배치 (무인 실행의 기본 — 미조사 전량을 매주 조금씩) ⭐
자재 종합 DB(material-research)의 **미조사 자재(`source:"seed"`) 전량(현재 314건)이 조사 계획(큐)에 올라가 있고, 매주 25건씩 소화**한다. 진행 현황 대시보드: `http://localhost:3011/research/`.
- **배치 가져오기**:
  - 로컬(서버 접근 가능): `GET http://localhost:3011/api/research/next?limit=25` → `data.items[]` (id·trade·category·subcategory·name·spec·brands·기준선 price 포함, `data.howto`에 반영 방법 동봉, `data.week`=이번 주차)
  - 클라우드/무인(localhost 불가): 레포 `material-research/data/materials.json`에서 `source==="seed"` 항목을 **공종 표준순(server.js 상단 `TRADE_ORDER`) → category → subcategory → name(한국어 locale)** 로 정렬해 상위 25건 — 서버 큐와 동일한 순서다.
- 각 항목의 `name + spec (+ brands)` 를 검색 키워드로 출처 3곳 이상에서 단가 조사한다(아래 카테고리↔출처 표는 힌트).
- 결과 반영은 4-3 의 material-research import로: **항목 `id`를 받은 그대로 유지**하고 `price:{min,max}`(관측 범위), `collected_date`(실제 수집일 YYYY-MM-DD), `notes:"출처: ○○, △△ / 수집일"` 을 담아 보낸다. **가격이 기준선과 동일해도 '검증 승격'으로 큐에서 빠지므로 조사한 항목은 빠짐없이 전송**한다(미확인 항목만 제외).
- 대기 전량 소화 후에는 API가 `mode:"refresh"` 로 수집일 오래된 순 재검증 회전을 자동으로 이어간다.

### 기본 카테고리 세트 (대화형 기본값 · 큐 조회 실패 시 폴백)
바닥재 · 벽/도배·도장 · 천장 · 몰딩 · 도어/창호 · 주방 · 욕실/위생도기 · 조명

## 절차 (이 순서대로 수행)

### 1) 조사 대상 확정
- **대화형**이면: 어떤 자재 카테고리인지, 비교할 자재(키워드)가 있는지 **먼저 묻는다**. (예: "이번엔 어떤 자재 단가를 볼까요? 바닥재 / 도장 / 창호 …")
- **무인(cron)**이면 묻지 않고 **리서치 큐의 다음 배치 25건**을 가져온다(위 "리서치 큐 배치"). 큐 조회가 불가능하면 기본 카테고리 세트로 폴백.

### 2) 브라우저 MCP로 출처 3곳 이상 탐색  ⭐
- 연결된 브라우저 MCP(크롬 확장 또는 Playwright) 중 가능한 것으로, 카테고리에 맞는 출처를 **3곳 이상** 연다.
- 각 출처에서 자재명·규격·단가(원)와 시세 동향을 수집한다. 가능하면 가격비교 페이지를 우선.
- **단가는 반드시 출처와 수집일을 함께 기록**한다. 모르면 추정하지 말고 "확인 불가"로 표기.
- 브라우저 MCP가 연결돼 있지 않으면: 그 사실을 먼저 알리고, 수동 연결을 안내한 뒤 중단한다(가짜 단가를 지어내지 않는다).

### 3) 단가 비교 md 정리 (정해진 포맷)
수집일은 Bash `date +%F`로 실제 값을 구한다. 카테고리별로 아래 포맷의 `자재단가-{카테고리}-{날짜}.md`를 만든다:

```markdown
# {카테고리} 자재 단가 리서치

- 수집일: {YYYY-MM-DD}
- 키워드: {자재1}, {자재2}, …

## 단가 비교
| 자재명 | 규격 | 단가 | 전주 대비 | 출처 |
|---|---|---|---|---|
| 강마루 | 6T, 시공/평 | ₩45,000 | ▲ +2.3% | 네이버쇼핑 |
| 포세린타일 | 600각/장 | ₩3,500 | = 보합 | 오늘의집 |
| … | … | … | 신규 | … |

> 가격 비교 요약 — 최저가: {출처} (₩…) · 평균가: ₩…

## 출처별 핵심
### {출처1}
- (시세 동향 3줄)
### {출처2}
- (시세 동향 3줄)
### {출처3}
- (시세 동향 3줄)

## 한 줄 요약
{이번 주 단가 동향 한 문장}

## 다음 액션
- [ ] {예: 을지로 ○○상사 실측 견적 요청}
- [ ] {예: 포세린 타일 대량 발주 단가 협의}
```

- **전주 대비**: 같은 카테고리의 직전 리서치 md(또는 노션 이전 기록)가 있으면 단가를 비교해 ▲상승/▼하락/=보합·%로 표기. 첫 회는 "신규".
- 표는 마크다운 테이블 문법, 체크박스는 `- [ ]`.

### 4) 노션 업로드 (Notion MCP)
- Notion MCP로 **"자재 단가 리서치"** 데이터베이스/페이지를 찾아(`notion-search`), 그 안에 이번 회차 페이지를 새로 만든다(`notion-create-pages`). 제목: `{카테고리} 자재 단가 — {YYYY-MM-DD}`.
- 위 md 본문을 노션 블록으로 올린다(표 포함). 여러 카테고리면 카테고리별 페이지 또는 한 페이지에 섹션으로.
- 업로드된 노션 페이지 URL을 결과에 함께 보고한다.
- ⚠️ **대화형 첫 실행**은 노션에 올리기 전 한 번 확인을 받는다. 무인(cron)은 사전 승인된 목적이므로 바로 업로드하되, 실패 시 로컬 md만 남기고 사용자에게 알린다.

### 4-2) interior-cost 단가 카탈로그 반영 (연계 ⑧)
- 수집한 단가를 인테리어 현장관리 앱 **interior-cost 의 단가 카탈로그**(`interior_catalog`)에도 자동 반영해, 견적 작성 시 최신 시세를 끌어 쓸 수 있게 한다.
- 각 자재를 import 포맷으로 변환:
  `{ trade: 21공종 중 근접값(바닥/도장/도배/타일/목공/전기/금속/유리/주방/화장실셋팅 등), name: 자재명, unit: 단위(M2/EA/통/식 등), material_price: 재료비 정수원, product_name: 규격/제품명, vendor: 출처/업체, source: 'research', price_date: '{YYYY-MM-DD 수집일}' }`
- **`POST {INTERIOR_COST_URL}/api/catalog/import`** body `{items:[...]}` 로 일괄 전송. `{INTERIOR_COST_URL}` = 배포된 interior-cost 주소(미배포면 로컬 `http://localhost:3010`). 응답 `{imported, updated, total}` 을 보고에 포함.
- 서버가 **중복(trade+name+product_name 동일)은 가격·기준일을 update** 하므로, 매주 돌리면 카탈로그가 최신 단가로 누적·갱신된다(중복 폭증 없음).
- interior-cost 서버에 연결 못 하면(미배포/오프라인/네트워크오류) **이 단계만 건너뛰고** 노션·로컬 저장은 그대로 유지한다(리서치 결과 자체는 절대 잃지 않음).

### 4-3) material-research 자재 종합 DB 반영 (베이스라인 위 주간 누적)
사무실 **자재 종합 DB**(`material-research`, 별도 앱·PORT 3011, 베이스라인 314건)에도 이번 주 자재를 누적시킨다. 신규 자재·가격 변동만 **덮어쓰지 않고 changelog로** 쌓인다("모두 모아놓고 그 위에 매주 업데이트").
- 각 수집 자재를 material 포맷으로 변환:
  `{ id: 큐 배치 항목이면 받은 id 그대로(신규 발견 자재만 생략), trade: 21공종 근접값, category: 대분류(바닥재/타일/도장재/벽지/천장재/도어·창호/주방·가구/욕실·위생도기/조명/…), subcategory: 중분류, name: 자재/제품명, spec: 규격, unit: 단위(M2/EA/장/통/롤/식 등), brands: [브랜드…], price: { min, max }(관측 시세 범위; 단일값이면 min=max), grade: 보급|중급|고급(선택), use: 용도(선택), collected_date: '{YYYY-MM-DD 실제 수집일}', notes: '출처: ○○ / {YYYY-MM-DD 수집일}' }`
- 큐 배치 항목은 **가격이 기준선과 동일해도 전송**한다 — 서버가 '검증 승격'(seed→research)으로 처리해 큐에서 내리고, 이미 research인 항목은 `collected_date`만 갱신(신선도 회전)된다.
- 전송 — **환경에 따라 둘 중 하나**:
  - **로컬 실행(서버 접근 가능)**: `POST http://localhost:3011/api/materials/import` body `{ source:'research', note:'{YYYY-Www} 주간 단가', items:[…] }`. 응답 `{imported, updated, unchanged, total, changelogAdded}` 을 보고에 포함.
  - **클라우드/무인 실행(localhost 접근 불가)**: 같은 JSON body를 레포 파일 **`material-research/data/incoming/자재DB-{YYYY-MM-DD}.json`** 로 쓰고 `git add/commit/push`. 로컬 앱이 다음 부팅 또는 `POST /api/materials/sync` 때 자동 흡수한다(import 멱등이라 중복 안 생김).
- 매칭키 `id` → 없으면 `trade+name+spec` 이므로 **중복은 가격만 update**, 매주 돌려도 폭증하지 않는다.
- material-research 서버에도 못 붙고 레포에도 못 쓰면 **이 단계만 건너뛰고** 노션·로컬 저장은 유지한다(결과는 절대 잃지 않음).

### 4-4) 제품 라이브러리 반영 (브랜드 × 모델 × 사진 — material-research /products/)
큐 배치 자재를 조사하는 김에, 그 자재의 **대표 실제 제품 1~2개**(예: 신한벽지 스케치 15106-1)를 제품 라이브러리에도 누적한다. **모델명·특징은 실제 페이지에서 확인된 것만**(날조 금지), 이미지는 페이지의 직접 이미지 URL(og:image 등)만.
- 항목 포맷: `{ brand: '신한벽지', model: '품번/라인명(실확인)', name: 제품명, trade: 21공종, category/subcategory, features: [확인된 특징 3~6], description, spec, unit, price: {min,max}(관측가)|null, image_url: '직접 이미지 URL'|'', url: '확인한 판매/제품 페이지', collected_date: 'YYYY-MM-DD' }`
- 전송: 로컬이면 `POST http://localhost:3011/api/products/import` body `{ source:'research', note:'{YYYY-Www} 제품 수집', items:[…] }` (매칭키 brand+model 멱등 UPSERT, 서버가 이미지를 `data/product-images/`에 자동 캐시). 클라우드면 **`material-research/data/incoming/제품DB-{YYYY-MM-DD}.json`** 에 `{ kind:'products', source:'research', items:[…] }` 로 커밋(파일명 '제품' 접두사 또는 kind 필드로 라우팅됨).
- 실패 시 이 단계만 건너뛴다. 라이브러리 확인: `http://localhost:3011/products/`.

### 5) 로컬 저장 & 보고
- md는 현재 작업 폴더의 `research/`(없으면 생성)에 저장한다. 같은 이름이 있으면 `-2`, `-3`… (덮어쓰지 않음).
- 저장 경로 + 노션 URL + 핵심 단가 변동(▲▼)을 한눈에 보고한다.
- 큐 배치 실행이면 **큐 진행 상황**도 한 줄 보고: `이번 배치 n건 반영 → 조사완료 {researched}/{total} (대시보드 /research/)` — import 응답과 `GET /api/research/queue`의 totals로 계산.

## 🔁 주간 자동 실행 (cron)
- 이 스킬은 **매주 1회 예약 루틴(Claude Code 스케줄)**으로 무인 실행되도록 설계됐다. 예약 실행 시: 묻지 않고 기본 카테고리 세트 → 출처 탐색 → md → 노션 업로드까지 끝낸 뒤, 전주 대비 변동 요약만 남긴다.
- 앱(`week-5/quest/research-skill/index.html`)의 "주간 자동 리서치" 패널은 **페이지가 열려 있을 때만** 도는 브라우저 스케줄러이고, 실제 무인 자동화는 이 cron이 담당한다.

## 주의사항
- **단가는 출처·수집일을 반드시 명시**하고, 확인 못 한 값은 지어내지 않는다("확인 불가"로).
- 브라우저/노션 MCP 미연결 시 솔직히 알리고 멈춘다(가짜 결과 금지).
- 기존 md를 덮어쓰지 않는다.
- 정해진 md 포맷(제목/수집일/단가 비교 표/출처별 핵심/한 줄 요약/다음 액션)을 **매번 동일하게** 유지한다 — 이게 이 스킬의 핵심 가치다.
