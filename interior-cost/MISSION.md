## 📖 한 문단 정의

**안도공간의 노션은 잘 만들어졌다** — 프로젝트 DB 하나에 발주·회의·매뉴얼·출력인원·현장사진·지원업무가 전부 relation으로 붙어 있고, 현장 페이지 하나만 열면 그 현장의 전부가 보인다. **문제는 그 노션이 사장의 머릿속 구조를 그대로 옮긴 지도라서, 그 머릿속을 공유하지 않은 사람에게는 지도가 아니라 미로라는 것이다** — 현장에 상주하는 시공책임은 폰으로 7단계를 뚫고 들어가 데이터베이스 필터를 걸 시간이 없고, 새로 온 디자이너는 스케치업은 능숙해도 우리 회사 노션이 어떻게 조직됐는지는 모른다. 그래서 현장의 진실은 결국 **그 구조를 아는 한 사람에게만 모이고**, 나머지는 카톡으로 사진을 보내고 그 사람이 대신 옮겨 적는다. **안도 현장은 안도공간 멤버들이 쓰는 올인원 현장 관리 앱으로, 노션과 같은 데이터를 "구조를 몰라도 되는 형태"로 건넨다** — 로그인하면 탐색 없이 내 업무가 먼저 뜨고, 일정이 밀리면 뒤 공정이 알아서 밀리고, 영수증은 찍으면 비용이 된다. **자동화는 편의가 아니라 접근성이다 — 앱이 계산을 대신하는 만큼, 구조를 모르는 사람도 현장에 참여할 수 있다.**

## Problem Statement

인테리어 현장의 본질은 **변경**이다. 자재가 단종되고, 자재상이 하루 늦고, 클라이언트가 어제 한 말을 오늘 바꾼다.

**1막 — 흩어짐.** 일정표·카톡방·캐드·엑셀이 각자 다른 진실을 말했다. 공정 하나가 밀리면 네 곳을 손으로 고쳐야 했고, 그 중 하나는 반드시 빠졌다.

**2막 — 노션이 이겼다.** 우리는 이걸 노션으로 풀었다. 프로젝트 DB 하나에 발주서·회의·매뉴얼·출력인원·현장사진·지원업무·고객·AS가 relation으로 붙었고, 현장 페이지 하나만 열면 그 현장의 전부가 보인다. **1막의 문제는 실제로 풀렸고, 노션은 지금도 잘 돌아간다.** 이 앱은 흩어진 파일과 싸우는 게 아니다.

**3막 — 노션은 구조를 아는 사람의 것이다.** ← **진짜 문제**

노션 프로젝트 DB에 닿는 경로:
```
데이터베이스 → (무제) → (무제) → (무제) → (무제) → 2. 현장(프로젝트) → 프로젝트 DB
```
**7단계 깊이, 중간 5개는 제목조차 없다.** 그리고 모든 현장 페이지에는 이 안내가 박혀 있다:

> *"메뉴얼을 가져오기위해 유형과 프로젝트 DB 필터를 해당 프로젝트에 맞게 선택해주세요."*

**매뉴얼 하나 보려고 데이터베이스 필터를 직접 설정하라는 지시가, 모든 현장 페이지에 적혀 있다.** 이게 허들의 정체이고, 문서로 남은 자백이다.

이건 **툴 리터러시 문제가 아니다.** 우리 디자이너는 스케치업·캐드·포토샵을 능숙하게 쓴다. 문제는 **이 노션이 한 사람의 머릿속 구조라서, 그 머릿속을 공유하지 않으면 못 쓴다는 것**이다.

**결과:**
- 현장에 있는 **시공책임**은 폰으로 7단계를 뚫을 시간이 없다 → 사진을 카톡으로 보낸다 → **누군가 대신 옮겨 적는다.**
- 새로 온 **디자이너**는 회사 노션 구조를 모른다 → 매번 물어본다 → **물어볼 수 있는 사람이 병목이 된다.**
- 그래서 **현장의 진실은 구조를 아는 한 사람에게만 모인다.** 그 사람이 현장에 나가 있으면 회사는 그 현장을 모른다.

**그리고 여기서 "노션은 계산을 안 해준다"가 합류한다.** 노션에서 일정 하나가 밀리면 뒤 공정 날짜를 **사람이 전부 다시 계산**해야 한다. 그걸 할 줄 아는 사람은 **구조를 아는 사람뿐이다.** 견적 총액을 예산 칸에 옮기는 것도, 카드 내역을 현장별로 가르는 것도 마찬가지다.

> ### 💡 **자동화는 편의가 아니라 접근성이다.**
> 사람이 계산기 노릇을 해야 하는 시스템은, **계산할 줄 아는 사람만 쓸 수 있는 시스템**이다. 앱이 계산을 대신하는 만큼 진입 장벽이 내려가고, 그만큼 더 많은 사람이 현장에 참여할 수 있다. **이 앱에서 자동화와 접근성은 같은 것의 두 이름이다.**

## 🚦 전환 조건 — "앱이 어느정도 만들어지면"의 정의

**지금은 노션으로 운영합니다. 그건 실패가 아니라 정상적인 전환 전 상태입니다.**
다만 **"어느정도"가 정의되지 않으면 전환은 영원히 오지 않습니다** — 앱은 계속 좋아지고, 노션은 계속 돌아가고, 아무도 "이제 옮기자"고 말하지 않습니다. 그래서 여기에 못 박습니다.

### 전환 조건 = 기능 개수가 아니라 이 두 문장

1. **시공책임이 폰으로** — 현장 브리핑(주차·엘리베이터·소음 규칙)을 보고 → 사진 올리고 → 출력인원 찍고 → 영수증 찍는다. **탐색 없이.**
   - 🚧 막는 것: **현장 브리핑 칸이 없다** (C-1)
2. **디자이너·신입이 PC로** — 새 현장을 템플릿으로 세팅하고 → 자료·매뉴얼을 붙이고 → 견적을 뽑는다. **아무한테도 안 물어보고.**
   - 🚧 막는 것: **프로젝트 템플릿이 없다** (C-2) — 새 현장 = 빈 탭 13개

### 전환 시점

**이 둘이 되면 → 다음 신규 현장을 "앱 온리"로 시작합니다.**

- ❌ **진행 중인 현장(분당 한땀 성형외과, ~8/29)을 중간에 옮기지 않습니다.** 이중입력 기간을 만드는 것이 이 앱이 죽는 가장 흔한 경로입니다.
- ❌ **전환 전에 기능을 더 붙이는 것은 전환을 늦출 뿐입니다.** v27에서 멈추고 C-1·C-2만 합니다.
- ✅ 전환 후에도 **노션은 계속 씁니다** — 아래 B 항목은 노션이 더 잘합니다.

## 🗂 노션과의 관계 — 공존한다

**결정: 노션과 앱은 함께 간다.** "노션을 그만 쓴다"가 아니라 **"현장 실행을 앱이 가져온다"**입니다.
**노션 = 회사의 뇌** (지식·기획·영업·개인). **앱 = 현장의 손** (실행·계산·기록).

### A. 이관한다 — 현장에 매인 것 (앱에 이미 자리 있음)

| 노션 | 앱 | 상태 |
|---|---|---|
| 협력업체 | ⚙️ 거래처 | ✅ **283개 이관 완료** |
| 매뉴얼 DB | 📖 매뉴얼 | ✅ **43건 이관 완료** (+ 필터 설정 불필요 = **허들 제거**) |
| 프로젝트 DB (현장 6+) | 🗂 프로젝트 | ⬜ 자리 있음 · **데이터 0** |
| 발주서 DB | 🧾 발주 | ⬜ 자리 있음 (+ 자동생성·PDF·공유링크로 **우위**) |
| 회의내용 DB / AS DB | 📒 미팅·AS | ⬜ 자리 있음 |
| 출력인원 DB | 👷 출력인원 | ⬜ 자리 있음 (+ 일당→인건비 자동으로 **우위**) |
| 현장사진 DB | 📷 현장사진 | ⬜ 자리 있음 · **0장** |
| 이번주/다음주 지원업무 | 📅 일정 (지원 구분) | ⬜ 자리 있음 |
| 리드/고객 DB | ⚙️ 고객/리드 | ⬜ 자리 있음 |
| 개인별 법인카드 사용내역 | 💸 비용 (💳 카드) | ✅ **구조적 우위** — CODEF 자동 동기화 + 가맹점 학습 |

### B. 노션에 남긴다 — 현장에 안 매인 것 (**명시적 anti-scope**)

- **세컨드 브레인 OS** (목표 · 프로젝트 · 할일 · 영역/자원 · 노트) — 개인 지식관리. 앱의 "내 업무"는 *현장 할일*이지 인생 할일이 아니다.
- **블로그 데이터베이스**, **주거공간**, **성공하는 프로젝트를 위한 준비** — 마케팅·영업 자산
- **코엑스 리빙페어** 등 전시/이벤트
- **[연구용] 템플릿 · 프로젝트 이름 연구용 · Project Brief** — 연구·실험 노트

👉 **이건 노션이 더 잘한다.** 삼키는 순간 우리는 **노션을 못 만드는 노션**이 된다.

### C. 아직 자리가 없다 — **이게 이관을 막고 있는 진짜 이유**

노션 현장 페이지에는 있는데 앱에는 칸이 없는 것들. **1차 타겟 2명이 정확히 여기 걸려 있다.**

| # | 없는 것 | 노션 실물 (분당 한땀 성형외과) | 누가 막히나 | 우선 |
|---|---|---|---|---|
| **C-1** | 🅿️ **현장 브리핑** (자유서술) | *"기계식 주차장이라 트럭·SUV 불가 / 보양된 엘리베이터만 / 윗층 독서실이라 소음작업 오전 중 완료 / 종료 시 분리수거·차단기·수도"* | **시공책임** — 현장 나가는 사람이 **제일 먼저 봐야 할 것.** 이거 하나 때문에 노션을 연다 | 🔥 **최우선** |
| **C-2** | 📋 **프로젝트 템플릿** | 새 현장 = 템플릿 복제 → 8개 DB 뷰 즉시 세팅 | **디자이너·신입** — 앱은 새 현장 = **빈 탭 13개.** 시작 비용이 노션보다 비싸다 | 🔥 **최우선** |
| C-3 | 📝 주거 사전 질문지 → 현장 자동생성 | 질문지 DB가 주소·면적·건물종류·입주일을 rollup으로 공급 | 대표·PM (1차 타겟 아님) | 보류 |
| C-4 | 📄 공사내용 자유서술 | 프로젝트 페이지 상단 | 대표·PM | 보류 |

> 💡 **C-1·C-2가 이 문서의 실행 가능한 결론입니다.** 앱이 v27까지 와서도 실무에 못 들어간 이유는 기능이 부족해서가 아니라, **1차 타겟 두 명이 노션을 열 이유가 각각 딱 하나씩 남아서**입니다.

## 💡 해결 방법 — 구조를 몰라도 되게 만드는 두 축

### 축 1: 탐색을 없앤다 (구조를 모르는 사람도 자기 일에 닿는다)

| 노션 | 앱 |
|---|---|
| 7단계 뚫고 → 프로젝트 찾고 → 필터 걸고 → 내 것 찾기 | **로그인 = 🙋내 업무.** 탐색 0단계 (v25) |
| "매뉴얼 보려면 유형과 프로젝트 DB 필터를 선택하세요" | **현장별 매뉴얼이 이미 붙어 있음** (v25) |
| 금액 지우고 정리해서 카톡에 올리기 | **클라이언트 공유 링크** — 금액은 화이트리스트로 차단 |

### 축 2: 계산을 대신한다 (계산할 줄 몰라도 참여한다)

노션도 relation은 있습니다. **노션이 못 하는 건 이것들입니다:**

| 노션에서 (사람이) | 앱에서 (자동으로) |
|---|---|
| 견적 총액을 예산 칸에 옮겨 적기 | **견적 확정 → 현장 예산 자동 갱신** (`server.js:4715`) |
| 하루 밀림 → 뒤 공정 날짜 전부 재계산 | **일정 선후관계 → 연쇄 이동 (cascade)** (`interior_schedule_deps`) |
| 월말에 카드 내역 현장별 분류 | **카드 동기화 → 현장 배분 → 비용 자동 생성 + 가맹점 학습** (CODEF/KB) |
| 영수증 손으로 타이핑 | **사진 찍으면 → OCR이 항목·금액 기입** (gpt-4o 비전) |
| 스케치업 물량 → 견적서에 옮겨 적기 | **모델 → CSV → 견적 물량 + 공종 자동 매핑** (DLP-TAKEOFF v2) |
| 자재 단가 매번 찾아 입력 | **자재 코드(WD/TX/WP-####) → 견적 단가 자동 주입** |
| 원가계산서는 엑셀(.xlsm)로 나가서 | **앱 안에서 공종 21 · 3분할 · 산출식 → 제안가** |
| 일당 계산해서 인건비 적기 | **출력인원 → 일당 프리셋 → 인건비 자동** |

**이 두 표에 없는 기능은 노션에서 해도 됩니다.**

## 🚫 Anti-Scope

- ❌ **노션의 비-현장 영역** — 세컨드 브레인 OS(개인 목표·노트), 블로그 DB, 마케팅 콘텐츠, 전시/이벤트, 연구용 템플릿. **현장에 매이지 않은 것은 앱에 들어오지 않는다.**
- ❌ **기공·조공에게 앱을 요구하지 않는다** — 일용직에게 로그인·앱 설치·입력을 시키지 않는다. 출력인원은 **시공책임이 대신 입력한다.**
- ❌ **캐드/스케치업을 앱 안에서 편집** — 모델링은 스케치업이 한다. 우리는 **물량만 받는다**(v23 CSV 경계선).
- ❌ **카카오톡 자체의 대체** — 우리가 없애는 건 "카톡방에 파일 올리기"이지 대화가 아니다.
- ❌ **회계·세무 시스템** — 세금계산서 발행, 4대보험 신고, 급여 이체는 우리 일이 아니다.
- ❌ **범용 SaaS화 (지금은)** — **내부 채택이 0인 상태의 외부 배포는 아무 의미가 없다.**
- ❌ **모바일 네이티브 앱** — 현장에서 폰 브라우저로 충분.
- ❌ **턴키 ERP** — **현장 하나의 생애주기**가 경계다.

## 🧭 설계 원칙

**① 구조를 배우게 하지 마라.** 이 앱이 노션을 이기는 유일한 지점이다. "이 기능을 쓰려면 먼저 ~를 설정하세요"라는 문장이 UI에 등장하는 순간, 우리는 노션을 다시 만들고 있는 것이다. *(노션의 자백: "메뉴얼을 가져오기위해 유형과 프로젝트 DB 필터를 선택해주세요")*

**② 자동화는 접근성이다.** 계산을 앱이 가져올 때마다 진입 장벽이 내려간다. 새 기능의 질문은 "이 화면이 필요한가"가 아니라 **"이걸 자동화하면 못 쓰던 사람이 쓰게 되는가"**이다.

**③ 두 곳에 각각 적으면 실패다.** — 단, **전환 전 노션 병행은 예외**(의도된 상태). 전환 후에도 같은 데이터가 양쪽에 남아 있다면 그건 실패다.

**④ 클라이언트에게는 진실만, 전부는 아니게.** 클라이언트 페이지는 항상 최신이어야 하지만, 금액·마진·내부 메모는 화이트리스트로 원천 차단한다. 신뢰는 한 번의 유출로 끝난다.
