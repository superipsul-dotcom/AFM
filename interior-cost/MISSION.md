## 📖 한 문단 정의

**인테리어 현장의 모든 것은 매일 바뀌는데, 그 변화를 담는 그릇은 오랫동안 흩어져 있었다 — 일정표, 클라이언트 카톡방, 캐드 파일, 예산 엑셀.** 우리는 그걸 노션으로 한 번 모았고, 노션은 정말 잘 모아줬다 — 프로젝트 DB에 발주·미팅·AS·고객·매뉴얼이 전부 relation으로 붙어 있다. **하지만 노션의 relation은 값을 비출 뿐, 바꾸지 않는다** — 공정이 하루 밀려도 뒤 공정은 그 자리에 있고, 견적을 확정해도 예산은 사람이 옮겨 적고, 카드값은 월말에 사람이 분류한다. **안도 현장은 안도공간 멤버들이 쓰는 올인원 현장 관리 앱으로**, 노션이 만들어 둔 그 구조를 그대로 이어받되 **모으는 데서 멈추지 않고 계산까지 한다** — 견적을 확정하면 예산이 따라 바뀌고, 공정 하나를 밀면 뒤 공정이 연쇄로 밀리고, 카드를 긁으면 비용이 되고, 클라이언트는 로그인 없이 언제나 최신을 본다. **노션 프로젝트 페이지를 더 이상 열지 않는 날이, 이 앱이 성공한 날이다.**

## Problem Statement

인테리어 현장의 본질은 **변경**이다. 자재가 단종되고, 자재상이 하루 늦고, 클라이언트가 어제 한 말을 오늘 바꾼다.

**1막 — 흩어짐.** 일정표·카톡방·캐드·엑셀이 각자 다른 진실을 말했다. 공정 하나가 밀리면 네 곳을 손으로 고쳐야 했고, 그 중 하나는 반드시 빠졌다.

**2막 — 노션.** 우리는 노션으로 이걸 해결했다. 프로젝트 DB 하나에 발주서·회의·매뉴얼·출력인원·현장사진·지원업무·고객·AS가 relation으로 붙었고, 현장 페이지 하나만 열면 그 현장의 전부가 보인다. **1막의 문제는 실제로 풀렸다.**

**3막 — 노션이 못 하는 것.** 노션은 **모아주지만 계산하지 않는다.**

- 공정이 하루 밀리면 → 뒤 공정 날짜를 **사람이 전부 다시 계산**한다. relation은 이걸 안 해준다.
- 견적을 확정하면 → 총액을 **사람이 예산 칸에 옮겨 적는다.**
- 원가계산서(간접비·보험·일반관리 7%·디자인 5%·이윤 12%)는 → **엑셀(.xlsm)로 돌아가서 돌린다.**
- 카드를 긁으면 → 월말에 **사람이 내역을 현장별로 분류**한다.
- 스케치업에서 물량을 뽑으면 → **사람이 견적서에 옮겨 적는다.**
- 클라이언트에게 보여주려면 → 금액이 섞여 있으니 **사람이 따로 정리해서 카톡에 올린다.**

**진짜 문제는 "흩어짐"이 아니라 "사람이 계산기 노릇을 하는 것"이다.** 노션은 현장의 상태를 *보여주는* 데까지 왔고, 그 다음 한 칸 — *반응하는 것* — 을 못 넘는다. 그 한 칸이 이 앱의 존재 이유 전부다.

## 🗂 노션과의 관계 — 무엇을 가져오고, 무엇을 남기는가

**결정: "노션을 그만 쓴다" = "현장 운영에서 노션을 뺀다"이지, "노션에 있는 걸 전부 앱으로 옮긴다"가 아니다.**
현장에 매이지 않은 것까지 앱이 삼키면, 이 앱은 노션의 열등한 복제본이 된다.

### A. 이관한다 — 현장에 매인 것 (앱에 이미 자리 있음)

| 노션 | 앱 | 상태 |
|---|---|---|
| 협력업체 | ⚙️ 거래처 | ✅ **283개 이관 완료** |
| 매뉴얼 DB | 📖 매뉴얼 | ✅ **43건 이관 완료** |
| 프로젝트 DB (현장 6+) | 🗂 프로젝트 | ⬜ 자리 있음 · **데이터 0** |
| 발주서 DB | 🧾 발주 | ⬜ 자리 있음 (+ PDF·공유링크로 **우위**) |
| 회의내용 DB / AS DB | 📒 미팅·AS | ⬜ 자리 있음 |
| 출력인원 DB | 👷 출력인원 | ⬜ 자리 있음 (+ 일당→인건비 자동으로 **우위**) |
| 현장사진 DB | 📷 현장사진 | ⬜ 자리 있음 · **0장** |
| 이번주/다음주 지원업무 | 📅 일정 (지원 구분) | ⬜ 자리 있음 |
| 리드/고객 DB | ⚙️ 고객/리드 | ⬜ 자리 있음 |
| 개인별 법인카드 사용내역 | 💸 비용 (💳 카드) | ✅ **구조적 우위** — CODEF 자동 동기화 + 가맹점 학습 |

### B. 남긴다 — 현장에 안 매인 것 (**명시적 anti-scope**)

- **세컨드 브레인 OS** (목표 · 프로젝트 · 할일 · 영역/자원 · 노트) — 개인 지식관리. 앱의 "내 업무"는 *현장 할일*이지 인생 할일이 아니다.
- **블로그 데이터베이스**, **주거공간**, **성공하는 프로젝트를 위한 준비** — 마케팅·영업 자산
- **코엑스 리빙페어** 등 전시/이벤트
- **[연구용] 템플릿 · 프로젝트 이름 연구용 · Project Brief** — 연구·실험 노트

👉 **이건 노션이 더 잘한다.** 노션 구독은 유지하고, 노션을 "회사의 뇌"로 남긴다. 앱은 "현장의 손"이다.

### C. 아직 자리가 없다 — **이게 이관을 막고 있는 진짜 이유**

노션 현장 페이지에는 있는데 앱에는 칸이 없는 것들. **이게 없으면 사람들은 계속 노션을 연다.**

| 없는 것 | 노션 실물 (분당 한땀 성형외과) | 왜 치명적인가 |
|---|---|---|
| 🅿️ **현장 브리핑** (자유서술) | "기계식 주차장이라 트럭·SUV 불가 / 보양된 엘리베이터만 / 윗층 독서실이라 소음작업 오전 중 완료 / 종료 시 분리수거·차단기·수도" | **현장 나가는 사람에게 가장 중요한 정보.** 구조화가 안 되는 텍스트라 앱에 칸이 없다. 이거 하나 때문에 노션을 연다 |
| 📋 **프로젝트 템플릿** | 새 현장 = 템플릿 복제 → 8개 DB 뷰가 즉시 세팅됨 | 앱은 새 현장을 만들면 **빈 탭 13개**. 시작 비용이 노션보다 비싸다 |
| 📝 **주거 사전 질문지 → 현장 자동생성** | 질문지 DB가 주소·면적·건물종류·입주일을 **rollup으로 공급** | 앱은 사람이 현장 폼에 손으로 입력. **노션이 이기고 있는 유일한 자동화** |
| 📄 **공사내용 자유서술** | 프로젝트 페이지 상단 | 요약 탭에 자유 메모 자리 없음 |

> 💡 **C가 이 문서의 가장 실행 가능한 결론입니다.** 앱이 v27까지 와서도 실무에 안 들어간 이유는 기능이 부족해서가 아니라, **현장 사람이 노션을 열 이유가 딱 4개 남아서**입니다. 기능을 더 붙이는 것보다 이 4개를 없애는 게 먼저입니다.

## 💡 해결 방법 — "모았다"가 아니라 "계산한다"

노션도 relation은 있습니다. **노션이 못 하는 건 이것들입니다** — 각 항목은 예전에 "사람이 계산기 노릇 하던 일"입니다.

| 노션에서 (사람이) | 앱에서 (자동으로) |
|---|---|
| 견적 총액을 예산 칸에 옮겨 적기 | **견적 확정 → 현장 예산 자동 갱신** (`server.js:4715`) |
| 하루 밀림 → 뒤 공정 날짜 전부 재계산 | **일정 선후관계 → 연쇄 이동 (cascade)** (`interior_schedule_deps`) |
| 월말에 카드 내역 현장별 분류 | **카드 동기화 → 현장 배분 → 비용 자동 생성 + 가맹점 학습** (CODEF/KB) |
| 영수증 손으로 타이핑 | **사진 찍으면 → OCR이 항목·금액 기입** (gpt-4o 비전) |
| 스케치업 물량 → 견적서에 옮겨 적기 | **모델 → CSV → 견적 물량 + 공종 자동 매핑** (DLP-TAKEOFF v2) |
| 자재 단가 매번 찾아 입력 | **자재 코드(WD/TX/WP-####) → 견적 단가 자동 주입** (material-research 연동) |
| 원가계산서는 엑셀(.xlsm)로 나가서 | **앱 안에서 공종 21 · 3분할 · 산출식 → 제안가** |
| 금액 지우고 정리해서 카톡에 올리기 | **클라이언트 공유 링크 → 언제 열어도 최신, 금액은 화이트리스트로 차단** |

**이 표가 이 앱의 존재 이유 전부입니다.** 이 표에 없는 기능은 노션에서 해도 됩니다.

## 🚫 Anti-Scope (하지 않기로 한 것)

- ❌ **노션의 비-현장 영역** — 세컨드 브레인 OS(개인 목표·노트), 블로그 DB, 마케팅 콘텐츠, 전시/이벤트, 연구용 템플릿. **현장에 매이지 않은 것은 앱에 들어오지 않는다.** 이걸 삼키는 순간 우리는 노션을 못 만드는 노션이 된다.
- ❌ **캐드/스케치업을 앱 안에서 편집** — 모델링은 스케치업이 한다. 우리는 **물량만 받는다**(v23 CSV 경계선).
- ❌ **카카오톡 자체의 대체** — 우리가 없앤 건 "카톡방에 파일 올리기"이지 대화가 아니다. 잡담·급한 연락은 계속 카톡에서.
- ❌ **회계·세무 시스템** — 세금계산서 발행, 4대보험 신고, 급여 이체는 우리 일이 아니다. 우리는 "이 현장에 얼마 썼나"까지.
- ❌ **범용 SaaS화 (지금은)** — 안도공간 워크플로에 딱 맞춘 것이 강점이다. **내부 채택이 0인 상태의 외부 배포는 아무 의미가 없다.**
- ❌ **모바일 네이티브 앱** — 현장에서 폰 브라우저로 충분.
- ❌ **턴키 ERP** — 인사·급여·전자결재로 번지지 않는다. **현장 하나의 생애주기**가 경계다.

## 🧭 설계 원칙

**두 곳에 각각 적는 순간, 그건 이 앱의 실패다.** — 그리고 **지금 우리가 그 상태다.** 앱과 노션에 같은 현장이 따로 존재하는 한, 이 앱은 문제를 푼 게 아니라 문제에 한 줄 더한 것이다.

**클라이언트에게는 진실만, 전부는 아니게.** 클라이언트 페이지는 항상 최신이어야 하지만, 금액·마진·내부 메모는 화이트리스트로 원천 차단한다. 신뢰는 한 번의 유출로 끝난다.

## ❓ Open Questions

- **[지금 가장 중요] 분당 한땀 성형외과를 앱으로 옮길 것인가, 다음 현장부터 할 것인가?**
  진행 중(~8/29)인 현장을 중간에 옮기는 건 이중입력 기간을 만든다. **다음 신규 현장을 "앱 온리"로 시작하는 쪽을 권합니다** — 노션 병행 없이, 템플릿부터 앱에서. 그게 유일하게 정직한 테스트다.
- **[선행 조건] 이관 잔여 항목 C 4개를 먼저 없앨 것인가?**
  특히 **현장 브리핑(주차·엘리베이터·소음 규칙)** 과 **프로젝트 템플릿**. 이 둘이 없으면 첫 현장에서 사람들이 노션으로 돌아간다. **기능 추가보다 이게 먼저입니다.**
- **[검증 필요] 현장 멤버가 실제로 폰으로 그 자리에서 입력하는가?**
  출력인원·사진·영수증은 **현장에서 즉시** 입력될 때만 의미가 있다. 사무실에서 몰아 입력하면 노션과 똑같아지고, 그러면 사람들은 익숙한 노션을 쓴다.
- **[정리] 노션 주간 자재 단가 리서치 cron** (`research` 스킬 → 노션, 월 09:00 KST)의 목적지를 material-research 앱(`/research/` 대시보드)으로 돌릴 것인가?
- **[범위] 노션 구독을 아예 해지할 것인가?** — **권장하지 않음.** 위 B 항목(세컨드 브레인·블로그·마케팅)은 노션이 더 잘한다. "현장 운영에서 노션을 뺀다"와 "노션을 없앤다"는 다른 결정이고, 후자는 이 앱을 망치는 경로다.
- **[이름] `interior-cost` 리네임 시점** — 라이브 URL·북마크 비용 vs 이름이 앱을 오해시키는 비용.
- **[확장] 안도 드라이브 연동** — `week-7/tesla-nav`가 이 앱의 현장·일정을 읽는다. 단, **앱에 실제 현장이 들어온 뒤에야 의미가 있다.** (지금 읽으면 "판교_사옥인테리어"로 안내한다.)
